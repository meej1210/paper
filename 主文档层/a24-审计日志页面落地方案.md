# 审计日志页面落地方案

> 适用项目：`D:\ziliao\dev`  
> 目标页面：`/audit-logs`  
> 目标定位：记录“谁在什么时候，对哪个扫描/报告/账号做了什么操作，结果如何”，与管理看板形成互补。

## 1. 页面定位

审计日志不是运行日志，也不是任务列表的复制品。它只记录平台中的关键安全操作和报告访问行为。

- 管理看板：看所有用户做了什么扫描，并进入扫描报告。
- 审计日志：看所有用户做了什么操作，并追溯操作证据。

审计日志页面要回答四个问题：

```text
谁
什么时候
对哪个资源
做了什么，结果如何
```

## 2. 页面最终效果

页面标题：

```text
审计日志
```

页面副标题：

```text
追踪登录、扫描、报告访问、导出和 DAST 授权行为。
```

页面结构：

```text
[顶部统计卡]
今日操作 / 登录失败 / 创建扫描 / 报告访问 / 报告导出 / DAST 授权

[图表区]
左：事件类型饼图
右：最近 24 小时操作柱状图

[筛选区]
用户关键词 / 动作类型 / 结果 / 时间范围 / 任务编号

[审计表格]
时间 | 用户 | 角色 | 动作 | 资源 | 结果 | IP | 详情

[展开详情]
展示结构化 JSON 摘要；如果关联任务，提供“查看任务报告”入口。
```

## 3. 事件类型设计

### 3.1 必须记录的事件

| 事件码 | 中文名称 | 触发场景 | 重要性 |
|---|---|---|---|
| `AUTH_LOGIN_SUCCESS` | 登录成功 | 用户登录成功 | 高 |
| `AUTH_LOGIN_FAILED` | 登录失败 | 用户名或密码错误 | 高 |
| `AUTH_LOGOUT` | 退出登录 | 用户主动退出 | 中 |
| `TASK_CREATE` | 创建扫描任务 | 创建 SAST / DAST / SCA 任务 | 高 |
| `TASK_CANCEL` | 取消任务 | 用户取消排队或运行中的任务 | 中 |
| `TASK_RERUN` | 重新执行任务 | 用户从已有任务发起复测 | 中 |
| `TASK_STATUS_CHANGE` | 任务状态变更 | 后台任务成功、失败、超时、取消 | 中 |
| `REPORT_VIEW` | 查看报告 | 用户进入任务报告页 | 高 |
| `REPORT_DOWNLOAD_JSON` | 下载 JSON 报告 | 用户下载原始 JSON 报告 | 高 |
| `REPORT_EXPORT_HTML` | 导出 HTML 报告 | 用户导出 HTML 报告 | 高 |
| `REPORT_EXPORT_PDF` | 导出 PDF 报告 | 用户导出 PDF 报告 | 高 |
| `DAST_AUTH_CONFIRMED` | DAST 授权确认 | 用户确认本地靶场或授权范围 | 高 |
| `DAST_AUTH_REJECTED` | DAST 授权拒绝 | 公网或未授权 DAST 目标被拒绝 | 高 |
| `ADMIN_DASHBOARD_VIEW` | 访问管理看板 | 管理员打开管理看板 | 中 |
| `ADMIN_TASK_VIEW` | 管理员查看用户任务 | 管理员查看其他用户扫描报告 | 高 |

## 4. 数据字段设计

### 4.1 审计日志表格字段

| 字段 | 来源 | 展示方式 |
|---|---|---|
| 时间 | `audit_logs.created_at` | `YYYY-MM-DD HH:mm:ss` |
| 用户 | 关联 `users.username` / `users.email` | 用户名 + 邮箱 |
| 角色 | 关联 `users.role` | 管理员 / 普通用户 |
| 动作 | `audit_logs.action` | 中文事件名 |
| 资源 | `resource_type` + `resource_id` | 任务 #54 / 用户 #28 / 报告 |
| 结果 | `detail.result` 或推断 | 成功 / 失败 / 拒绝 |
| IP | `audit_logs.ip_address` | 原样展示 |
| 详情 | `audit_logs.detail` | 默认摘要，点击展开 JSON |

## 5. detail 内容规范

`detail` 必须尽量使用 JSON 字符串，不要混用纯文本和 JSON。前端收到后：

1. 能解析 JSON：展示摘要 + 可展开 JSON。
2. 不能解析 JSON：按普通文本展示。

## 6. 前端页面方案

### 6.1 顶部统计卡

放 6 个小卡，不做大英雄区：

```text

创建扫描
报告访问

```

要求：

- 卡片高度不超过 90px。
- 数字字号控制在 24px-30px。
- 背景使用浅色，文字深色。
- 失败类指标使用红色强调。

### 6.2 图表区

两张图并排：

1. 事件类型饼图
   - 认证登录
   - 扫描任务
   - 报告访问
   - 报告导出
   - DAST 授权
   - 管理员操作

2. 最近 24 小时操作柱状图
   - 横轴：小时
   - 纵轴：操作数量
   - 鼠标悬浮显示数量，可先不做 tooltip

不引入新图表库，使用 SVG + CSS 实现。

### 6.3 筛选区

筛选项：

| 筛选项 | 控件 | 参数 |
|---|---|---|
| 用户关键词 | input | `keyword` |
| 动作类型 | select | `action` |
| 结果 | select | `result` |
| 任务编号 | input | `resource_id` |
| 开始时间 | datetime-local | `start_time` |
| 结束时间 | datetime-local | `end_time` |

按钮：

```text

重置
```

### 6.4 日志表格

表头：


行内展示：

- 动作用彩色标签。

## 7. 后端接口落地方案

### 7.1 修改接口

当前已有：

```text
GET /api/audit-logs
```

建议增强为：

```text
GET /api/audit-logs?page=1&page_size=20&keyword=&action=&result=&resource_id=&start_time=&end_time=
```

返回内容增加：

```text
items
pagination
summary
charts
```

### 7.2 查询逻辑

- 普通用户：只看自己的审计日志。
- 管理员：看全平台审计日志。
- 默认按 `created_at desc` 排序。
- `page_size` 最大 100。
- `keyword` 匹配用户名、邮箱、动作、详情。
- `action` 精确匹配事件码。
- `result` 从 `detail.result` 解析；如果无法解析，可先用 `detail LIKE` 实现第一版。
- `resource_id` 精确匹配。
- 时间范围使用 `created_at`。

### 7.3 summary 计算

以当前筛选范围或最近 24 小时为准，第一版建议用“今日”统计：

```text
today_count：今日全部日志数
login_failed_count：今日登录失败数
task_create_count：今日创建任务数
report_view_count：今日查看报告数
report_export_count：今日导出报告数
dast_authorized_count：今日 DAST 授权确认数
```

### 7.4 charts 计算

`by_action`：

```json
{
  "AUTH_LOGIN_SUCCESS": 8,
  "TASK_CREATE": 5,
  "REPORT_VIEW": 7
}
```

`by_hour_24h`：

```json
[
  { "hour": "00:00", "count": 0 },
  { "hour": "01:00", "count": 0 },
  { "hour": "02:00", "count": 1 }
]
```

## 9. 安全与隐私要求

审计日志禁止记录：

- 明文密码
- JWT Token
- Cookie
- 上传文件完整内容
- 大段源码内容
- 过长的扫描原始报告

审计日志可以记录：

- 用户 ID
- 用户名 / 邮箱
- 任务 ID
- 扫描类型
- 扫描目标名称
- DAST 目标 URL / 主机 / IP / 授权策略
- 报告导出格式
- 操作结果
- IP 地址

## 13. 分阶段实施建议

## 14. 最终验收标准

完成后，审计日志页面应满足：

- 能看到所有关键安全操作。
- 管理员能看到所有用户操作。
- 普通用户只能看到自己的操作。
- 能通过筛选快速定位某个用户、任务、动作。
- 能从报告访问日志跳回对应报告。
- 能展示事件类型分布和时间趋势。
- 不记录密码、Token 等敏感信息。
- 页面和管理看板职责清晰，不重复。
