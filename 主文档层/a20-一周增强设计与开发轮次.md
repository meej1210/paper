# DevSecOps 平台一周增强设计与开发轮次

> 目标：把当前系统从“覆盖开题的可运行原型”推进到“论文与答辩更有说服力的增强版”。本文按独立开发轮次拆分，每一轮都能交给下一个 AI 单独执行、测试和交付。
>
> 当前基线：`D:\ziliao\dev` 已有 Flask + SQLAlchemy + Celery + Redis + Vue3/Vite/TypeScript 主链路；已有 SAST（Bandit / Semgrep）、DAST（Wapiti）、任务管理、审计日志、报告导出、AI 解读、基础 DAST 白名单配置和真实 async 启动脚本。
>
> 一周策略：优先做“论文可写、答辩可演示、改动可控”的增强，不引入 CodeQL、K8s、完整 Jenkins 等高风险重工程。

## 1. 总体范围

### 1.1 本周要解决的五类不足

| 方向 | 本周目标 | 最终论文表述 |
|---|---|---|
| SCA 依赖漏洞扫描 | 增加面向 `requirements.txt` 的 Python 依赖漏洞扫描任务 | 平台覆盖代码缺陷、运行态漏洞和依赖漏洞三类风险 |
| DAST 授权边界 | 将已有 URL 限制升级为“授权确认 + 白名单策略 + 审计证据” | 动态扫描仅面向本地靶场或明确授权目标，降低误用风险 |
| CI/CD 集成 | 提供可运行脚本和示例流水线，而非搭建重型平台 | 平台具备流水线质量门禁接入能力 |
| 有限并发与队列稳定性 | 明确 Celery 队列、接口限流、任务超时和并发实验 | 系统支持教学 / 中小项目场景下的有限并发与排队处理 |
| 管理员审计看板 | 管理员登录后看到审计、任务、风险与运行状态汇总 | 平台具备基础安全运营与审计追踪能力 |

### 1.2 推荐开发顺序

1. 第 1 轮：SCA 依赖漏洞扫描模块。
2. 第 2 轮：DAST 授权边界增强。
3. 第 3 轮：CI/CD 集成脚本与示例流水线。
4. 第 4 轮：有限并发、队列稳定性与实验脚本。
5. 第 5 轮：管理员审计看板与统计接口。
6. 第 6 轮：论文证据沉淀、演示脚本与验收清单。

这个顺序的原因：SCA 是功能增量；DAST 授权是安全合规增量；CI/CD 和并发主要是工程化证明；管理员看板最后做，避免前面数据结构还未稳定时重复改 UI。

## 2. 第 1 轮：SCA 依赖漏洞扫描模块

### 2.1 目标

新增一种任务类型 `SCA`，扫描用户上传的 Python 依赖文件或代码包中的 `requirements.txt`，调用 `pip-audit` 输出 JSON，解析依赖漏洞并入库，前端可创建任务、查看结果、下载报告。

优先选择 `pip-audit`，理由是它可以审计 Python 环境、requirements 文件和依赖树，支持 `-r requirements.txt` 与 JSON 输出；Safety 也支持扫描和 JSON 输出，但新版本更偏向整目录扫描和平台化流程，作为本项目一周增强的首选成本更高。

### 2.2 范围

必须完成：

- 后端新增 `TaskType.SCA`。
- 新增 SCA 创建接口：`POST /api/sca/tasks`。
- 支持上传 `.txt`、`.zip`；`.txt` 直接视为 requirements 文件，`.zip` 解压后查找根目录或一级目录下的 `requirements.txt`。
- 后端调用：`pip-audit -r <requirements.txt> --format=json --output <report_path>`。
- 解析字段：包名、当前版本、漏洞 ID、别名、修复版本、描述。
- 新增 SCA 结果和问题表。
- 前端统一扫描页增加 `SCA` 类型入口。
- 任务列表、详情页显示 `SCA` 标签、漏洞数量、修复版本建议。
- JSON 下载和 HTML/PDF 导出复用现有报告导出框架。
- 测试覆盖：参数校验、成功解析、无漏洞、扫描命令失败。

不做：

- 不做自动修复依赖版本。
- 不做 NPM/Maven/Go 多语言 SCA。
- 不做 SBOM 生成。

## 4. 第 3 轮：CI/CD 集成脚本与示例流水线

### 4.1 目标

不搭建重型 CI/CD 平台，而是提供“脚本 + 示例流水线 + 质量门禁退出码”。这样论文可以证明平台具备被流水线调用的能力。

### 4.2 范围

必须完成：

- 新增本地 CI 安全扫描脚本。
- 脚本支持登录、创建 SAST 或 SCA 任务、轮询状态、下载报告。
- 支持质量门禁：发现高危或可修复依赖漏洞时返回非零退出码。
- 新增 GitHub Actions 示例文件或文档片段。
- 新增 CI 集成说明文档。

不做：

- 不要求远程 GitHub 仓库真实运行。
- 不搭 Jenkins/GitLab Runner。
- 不做多项目流水线编排。

### 4.3 文件设计

建议新增文件：

- 新增：`scripts/ci_security_scan.py`
  - 参数：`--base-url`、`--username`、`--password`、`--type sast|sca`、`--target-file`、`--fail-on high|fixable|any`、`--timeout`、`--out-dir`。
  - 流程：登录 -> 创建任务 -> 轮询 `/api/tasks` 或详情接口 -> 下载 JSON -> 计算门禁 -> 设置退出码。
- 新增：`scripts/ci_security_scan.ps1`
  - PowerShell 包装脚本，便于 Windows 演示。
- 新增：`.github/workflows/security-scan.example.yml`
  - 示例：启动服务后调用 `python scripts/ci_security_scan.py`。
- 新增：`主文档层/a21-CI-CD集成说明.md`
  - 包含本地运行命令、流水线示例、质量门禁说明、截图位置。

### 4.4 脚本行为

推荐脚本逻辑：

1. `POST /api/auth/login` 获取 token。
2. 根据 `--type` 调用：
   - SAST：`POST /api/sast/tasks`
   - SCA：`POST /api/sca/tasks`
3. 每 3 秒轮询任务详情。
4. 状态为 `SUCCESS` 时下载 JSON 报告。
5. 状态为 `FAILED/TIMEOUT/CANCELLED` 时退出码为 2。
6. 如果命中质量门禁，退出码为 3。
7. 正常通过退出码为 0。

### 6.2 范围

必须完成：

- 管理员专属路由 `/admin`。
- 后端管理员统计接口。
- 前端管理员看板页面。
- 审计日志页保留，作为看板的下钻入口。
- 非管理员访问 `/admin` 时显示无权限或跳转。

不做：

- 不做复杂用户管理。
- 不做图表库大引入，优先使用 CSS 条形图 / 卡片。
- 不做跨组织多租户。

### 6.3 后端设计

建议新增或修改文件：

- 新增：`backend/app/api/admin.py`
  - `GET /api/admin/dashboard`
  - `@jwt_required()` + `@require_roles("admin")`
  - 返回结构：
    - `task_totals`：总数、SAST、DAST、SCA、成功、失败、运行中、排队中。
    - `risk_summary`：SAST 高 / 中 / 低、DAST 高 / 中 / 低、SCA 漏洞数、可修复数。
    - `recent_audit_logs`：最近 10 条。
    - `recent_failed_tasks`：最近 10 条失败 / 超时任务。
    - `dast_targets`：最近 DAST 目标、授权状态、策略。
    - `runtime`：后端健康、celery 模式、redis 状态、dast scanner 状态。

  - 注册 `admin_bp`，前缀 `/api/admin`。
- 复用：`backend/app/api/health.py` 中的健康检查逻辑，必要时抽到 service，避免复制太多。

## 9. 风险控制

- 如果 `pip-audit` 安装或网络查询不稳定：保留 SCA 模块结构，测试中 mock 命令输出；演示时使用预置 JSON 报告作为降级方案。
- 如果 SCA 前端时间不足：先复用任务详情页的通用报告区，保证后端能力和报告导出。
- 如果管理员看板时间不足：优先做后端 `/api/admin/dashboard` 和简洁卡片，不引入复杂图表。
- 如果并发实验数据不理想：论文中如实写“有限并发验证”，强调教学实验环境定位，不宣称生产级高并发。
- 如果迁移风险高：每轮先备份数据库，迁移脚本小步执行，必要时在测试数据库中验证后再跑演示库。

## 10. 给下一位 AI 的执行规则

1. 每次只执行一个轮次，不要同时大改五个方向。
2. 每轮开始先读本文件和当前相关代码。
3. 每轮先写 / 改测试，再改实现。
4. 每轮结束必须运行后端测试和前端构建。
5. 每轮结束更新对应主文档和验收记录。
6. 不要移除现有 SAST/DAST/AI/报告功能。
7. 不要把普通用户任务范围扩大为全局；全局统计只放在管理员视图。
8. 不要引入 CodeQL、Jenkins、K8s 等重型依赖，除非用户另行确认。
