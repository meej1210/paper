## 1. 使用目标

本脚本用于答辩、联调和现场演示，目标是稳定走通以下主链路：

登录 -> 创建 SAST -> 查看结果 -> 导出报告 -> 创建 DAST -> 查看结果 -> 查看审计日志

建议总时长控制在 8~12 分钟，优先展示“系统已经能跑通”，而不是在现场等待完整扫描。

## 3. 推荐演示顺序

### 3.1 登录
- 进入前端首页
- 使用已准备账号登录
- 说明平台已具备鉴权与角色控制
- 如需自然过渡，可顺带说明：未登录访问受保护页面会跳转 `/login`

### 3.2 SAST 演示
- 上传 `tests` 中的示例文件或压缩包
- 创建 SAST 任务
- 展示任务状态、问题统计、重点问题与 AI 解读
- 演示 JSON / HTML / PDF 导出

推荐讲解点：

- SAST 不是占位接口，而是实际调用 Bandit
- 支持文件与压缩包上传
- 结果会落库并生成报告
- 可从高危问题切入，再展示 AI 解读增强

如现场时间紧张，可直接打开已完成任务详情页，避免等待扫描。

### 3.3 DAST 演示
- 提交本地白名单目标（Juice Shop）
- 说明白名单与超时控制的安全限制
- 展示漏洞类型、风险等级与整改建议
- 演示导出结果

推荐讲解点：

- DAST 只针对本地或授权白名单目标，不是公网扫描器
- 超时参数与白名单校验是产品边界，不是缺陷
- 可重点展示漏洞分类、风险等级、整改建议与导出能力

### 3.4 审计日志演示
- 切换管理员视角
- 展示登录、创建任务、导出报告等关键操作已被记录

可过滤的维度包括：

- `user_id`
- `action`

演示时建议突出“关键行为可追踪、可回溯”。

### 3.5 收尾话术

- 当前系统已具备注册、登录、SAST、DAST、任务管理、审计日志与报告导出主链路
- 当前工作的重点不是补基础功能，而是稳定性、异常验证与实验材料沉淀

## 4. 故障应对口径

- 若 PDF 导出失败：优先解释为本机依赖问题，并转而展示 HTML / JSON 导出
- 若 DAST 超时：说明动态扫描对目标可用性和时间窗口依赖更强
- 若页面提示异常：优先回退到已成功任务进行展示，避免现场等待长时间扫描
- 若 Worker 未消费任务：优先展示已完成任务详情页，并说明真实 async 模式已在 2026-05-07 复测
- 若审计日志无法访问：确认当前用户是否为 `admin`

### 4.1 现场兜底策略

- 第一优先：展示已完成任务与现成报告
- 第二优先：展示前端页面、健康检查与历史联调记录
- 第三优先：展示 `tmp\run-control\logs` 中的启动成功证据
