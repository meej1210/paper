## 一、eager 模式与真实 worker 模式的边界

### 1. eager 模式是什么

当 `CELERY_TASK_ALWAYS_EAGER=true` 时，任务会在当前进程内同步执行，不依赖 Redis 队列，也不需要单独启动 Worker。

适合场景：

- `backend/tests` 自动化测试
- 后端接口与业务逻辑的快速回归
- 不关心真实异步链路时的本地开发验证

### 2. eager 模式能验证什么

eager 模式适合验证：

- 注册、登录、JWT 鉴权
- SAST / DAST 接口参数校验
- 任务状态流转与接口返回结构
- 结果入库与基本导出逻辑
- 业务逻辑层的回归测试

### 3. eager 模式不能替代什么

eager 模式不能替代以下验证：

- Redis 队列可用性
- Celery Worker 实际启动与消费任务
- API 与 Worker 进程间配合
- 真实扫描器调用、超时与失败处理
- HTML / PDF 导出在真实异步任务中的表现

### 4. 真实 worker 模式需要覆盖什么

真实 worker 模式至少应覆盖：

- `CELERY_TASK_ALWAYS_EAGER=false`
- Redis 正常启动
- API 与 Worker 同时在线
- 至少 1 组 SAST 成功样例
- 至少 1 组 DAST 成功样例
- JSON / HTML / PDF 导出可访问

### 5. 推荐结论

推荐采用“双轨验证”：

- 日常开发与回归：优先 `backend/tests` + 前端构建
- 联调、答辩、实验结论：必须补一轮真实 async 验证

## 二、开发 / 演示 / 实验三类环境建议

### 1. 开发环境
- 目标：快速迭代、快速回归
- 推荐：eager 模式，可选 SQLite 或本地 MySQL
- 重点：接口可用、页面可构建、基础功能不中断

### 2. 演示环境
- 目标：保证主链路稳定演示
- 推荐：Redis + API + Worker + Frontend + Juice Shop 全部拉起
- 重点：登录 -> SAST -> 导出 -> DAST -> 审计日志 主链路顺畅

### 3. 实验环境
- 目标：沉淀可写入论文 / 汇报的数据
- 推荐：保持扫描器、数据库与导出依赖稳定一致
- 重点：任务耗时、问题数量、严重度分布、类型分布等结果可复现

## 三、最小落地建议

- 平时开发：先跑 `backend/tests` 与 `npm run build`
- 对外演示前：至少跑一轮真实 async 联调
- 形成实验结论前：固定环境版本，避免结果漂移

## 一周增强验证边界

- SCA 边界：当前仅覆盖 Python `requirements.txt` 依赖漏洞识别，不覆盖 NPM、Maven、Go、SBOM 生成或自动修复。
- DAST 边界：动态扫描必须由用户确认授权，默认只允许 localhost、127.0.0.1、私有网络或配置白名单目标；不做企业审批流、公网域名所有权验证或 DNS 重绑定深度防护。
- CI/CD 边界：提供脚本和示例流水线，证明平台可被流水线调用；不搭建 Jenkins、GitLab Runner 或完整远程 CI 环境。
- 并发边界：通过 Celery/Redis、限流配置、统计接口和并发脚本证明有限并发能力，定位为教学实验与中小项目场景，不宣称生产级高并发。
- 管理员视图边界：全局统计仅在 `/admin` 管理看板展示，普通用户仍只查看自己的任务和报告。

## DAST 样本扩展边界

- 本地靶场边界：当前推荐使用 Docker 本地靶场扩充 DAST 样本，默认包括 Juice Shop、DVWA、WebGoat 和 Mutillidae II。
- 采样边界：批量采样脚本 `scripts/run_dast_lab_samples.py` 只读取显式配置的 targets 文件，并且创建任务时固定传入 `authorization_confirmed=true`，仍走后端 URL 校验。
- 公网边界：公网 URL 后续可以支持，但默认不开放任意公网扫描。公网目标必须加入 `DAST_ALLOWED_HOSTS`，或后续进入管理员授权目标表。
- 白名单边界：`DAST_ALLOW_PUBLIC_TARGETS=false` 时，只有 localhost、私网目标或 `DAST_ALLOWED_HOSTS` 中的主机可以创建 DAST 任务。
- 实验边界：论文和答辩中应把本地靶场样本与公网授权样本分开统计，避免把非授权公网目标写成正常扫描样本。
