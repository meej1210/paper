# 中期答辩汇报主稿（草稿）

这份文档按“明天就要答辩、要尽快做 PPT”来整理，目标是：

- 帮你明确汇报主线
- 告诉你每一页 PPT 讲什么
- 标出哪些地方适合放截图
- 标出哪些地方适合现场演示
- 尽量避开不稳定点，减少答辩翻车风险

---

## 一、先给结论：这次中期答辩应该怎么讲

### 1. 不要把重点放在“我写了很多代码”

中期答辩老师更关心的是：

- 你为什么做这个题目
- 你现在做到什么程度
- 这个系统是不是已经能跑起来
- 你是否已经形成了清晰的技术路线
- 接下来还能继续做什么

所以汇报主线建议是：

> 从需求出发 -> 给出系统设计 -> 展示当前实现成果 -> 用真实运行结果证明系统不是空壳 -> 最后说明剩余工作和后续计划

### 2. 你这次最有说服力的点，不是“概念”，而是“已经跑通”

目前你最强的答辩材料是：

- 前后端系统已经可运行
- SAST 已真实接入 Bandit
- DAST 已真实接入 Wapiti
- Celery + Redis 异步链路已经真实跑通
- 任务列表、详情、取消、重跑、审计日志已经有界面和接口
- 真实联调已经完成
- SAST / DAST 的 JSON 和 HTML 导出已经验证成功

这意味着你不应该把自己讲成“还在设计阶段”，而应该讲成：

> 我已经完成了一个可运行的阶段性原型，目前重点转向稳定性、实验数据和答辩材料沉淀。

### 3. PPT 风格建议

建议控制在 `10~12` 页左右，讲 `8~12` 分钟最稳。

推荐风格：

- 每页只讲一个核心问题
- 少放大段文字，多放结构图、流程图、系统页面截图
- 每页最多 3~5 个要点
- 有图就尽量用图，不要整页全是文字

## 二、推荐 PPT 结构（可直接照着做）

下面按“第几页 PPT”来写。

## 第 1 页：标题页

### 标题建议

`基于 DevSecOps 的自动化代码审计与漏洞扫描平台设计与实现`

### 副标题可写

- 中期答辩汇报
- 学生姓名 / 学号 / 指导老师
- 学院 / 专业
- 日期

### 这一页怎么讲

用 20~30 秒说明：

- 我的课题是什么
- 我的系统是做什么的
- 今天主要汇报目前已经完成的阶段成果

## 第 2 页：研究背景与课题意义

### 这一页讲什么

讲“为什么要做这个平台”。

### 可讲的 4 个点

- 传统安全审计依赖人工，效率低、重复工作多
- 开发阶段往往缺少自动化安全检查，漏洞发现滞后
- DevSecOps 强调把安全能力前移到开发与交付流程中
- 因此需要一个能统一管理 SAST / DAST、任务执行和结果展示的平台

### 这一页怎么讲

用一句话总结：

> 本课题希望把静态审计、动态扫描、任务调度和结果分析整合到一个平台中，形成可视化、可追踪、可复测的安全检测流程。

## 第4页：哪些还没做（当前主要问题）
- 前端还存在稳定性交付风险：`TaskDetailView.vue` 中异常字符会导致前端构建失败。
- 任务重跑后的详情跳转逻辑还不够稳：页面使用了 `name: "task-detail"` 跳转，但路由中未定义同名路由，存在跳转失效风险。
- 当前验证仍偏接口和单元 / 集成层，完整前后端联调验收、真实 Redis / Celery / 扫描器链路验收还需要进一步补齐。
- 文档分层与基线已经完成一轮整理，但恢复版文档与归档文档仍需继续精修，避免答辩表述出现“历史原文”和“恢复版结论”混用。
- 异常场景覆盖还不够完整，前端对 401 / 403 / 500、导出失败、AI 解读失败、任务取消失败等提示还需要加强。
- SAST / DAST 的边界异常验证还未系统化完成，例如非法文件、空文件、非法 URL、非白名单目标、超时、扫描器失败、报告缺失等。

## 第5页：哪些还没做（成果材料层面）
- 真实异步 worker 模式的联调记录还需要补齐，需明确哪些场景只在 eager 模式验证，哪些场景已经在真实 Worker 模式下跑通。
- 运维与演示一致性还需要继续强化，需补齐关键依赖检查机制，降低 Bandit、Wapiti、Redis、WeasyPrint 等环境问题带来的风险。
- 开发环境、演示环境、论文实验环境的配置差异还需要进一步说明，避免结果不一致。
- 面向论文和答辩的实验材料还不够完整，仍需补充任务耗时、问题数量、严重度分布、类型分布、整改前后对比等数据。
- 平台能力边界说明还需要沉淀清楚，例如 DAST 仅面向本地或授权目标，AI 解读属于增强能力，PDF 导出和扫描器能力存在环境依赖。
- 并发、性能、RBAC、仪表盘、实验统计页等增强项暂未作为主线完成项，属于有余力时再推进的内容。
